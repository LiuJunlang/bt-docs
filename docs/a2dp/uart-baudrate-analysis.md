# 蓝牙芯片 UART 波特率选择指南

在嵌入式蓝牙系统中，Host 与 Controller 之间通常通过 UART 接口传输 HCI 数据。对于 WiFi+BT Combo 芯片，UART 波特率的选择直接影响多设备并发场景下的音频传输稳定性。

本文从理论计算、编码格式对比、软件传输机制、多设备并发场景四个维度展开分析，并澄清一些常见的认知误区。

---

## 一、921600 波特率的理论吞吐量

### 1.1 基础计算

典型的 UART 配置为 **8 位数据、1 位起始位、1 位停止位、无校验（8N1）**，即每传输 1 字节需要 **10 个 bit**。

| 参数 | 数值 |
|------|------|
| 波特率 | 921,600 bps |
| 每字节 bit 数 | 10 bit（8N1） |
| **理论最大吞吐量** | **92,160 Bytes/s ≈ 90 KB/s** |

### 1.2 HCI 协议开销

A2DP 音频数据通过 HCI ACL 数据包传输，每层协议都会引入额外开销：

| 协议层 | 开销 | 说明 |
|--------|------|------|
| HCI ACL Header | 4 Bytes | Handle（2B）+ PB/BC + Length |
| L2CAP Basic Header | 4 Bytes | Length（2B）+ CID（2B） |
| **每包总开销** | **8 Bytes** | Basic Mode 下固定开销 |

当 L2CAP MTU 配置为 1021 字节时，每包开销占比为：

```
开销比例 = 8 / (1021 + 8) ≈ 0.78%
```

这个比例很小，但还需要考虑 HCI Command/Event 的穿插传输、以及包边界对齐等因素。综合估算，**实际可用作音频数据传输的带宽约为 80～85 KB/s**。

> **关键认知**：这里的 "80～85 KB/s" 是理论上限，前提是 UART 发送端能够**持续不断地**向硬件 TX 寄存器提供数据，线路上没有空闲间隙。

## 二、常见 A2DP 编码格式的带宽需求

### 2.1 各编码格式对比

以下列出典型"音乐级"质量配置下的实时码率需求：

| 编码格式 | 典型码率 | 音频数据净荷（KB/s） | 加 HCI 开销后（约） | 与 921600 带宽对比 |
|----------|----------|----------------------|---------------------|-------------------|
| SBC（高品质） | 328 kbps | 41 | ~45 KB/s | ✅ 充足，余量大 |
| AAC | 256 kbps | 32 | ~35 KB/s | ✅ 非常轻松 |
| aptX | 352 kbps | 44 | ~48 KB/s | ✅ 够用，有余量 |
| aptX HD | 576 kbps | 72 | ~78 KB/s | ⚠️ 接近极限 |
| LDAC（中档 660kbps） | 660 kbps | 82.5 | ~90 KB/s | ❌ 超出理论上限 |
| LDAC（最高档 990kbps） | 990 kbps | 123.75 | ~135 KB/s | ❌ 远超上限 |

### 2.2 初步结论

从纯算术角度看：

- **SBC / AAC / aptX**：所需带宽远低于 921600 的理论上限，**完全没问题**。
- **aptX HD**：所需带宽 ~78 KB/s，已接近 80～85 KB/s 的实际可用上限。在系统负载较低时可用，但余量很小。
- **LDAC 660 kbps**：需要 ~90 KB/s，**已超出 921600 的理论极限**（即使不考虑任何开销，纯数据也才 90 KB/s）。
- **LDAC 990 kbps**：需要 ~124 KB/s，**远超 921600 上限**，必然卡顿。

> **注意**：以上计算假设 UART 能够**全速连续传输**。如果软件侧无法持续供数，有效吞吐量会进一步下降，详见下一节。

---

## 三、软件传输机制对有效吞吐量的影响

### 3.1 DMA vs CPU-driven 传输

UART 数据的搬运方式直接决定了有效吞吐量：

| 传输方式 | 机制 | CPU 占用 | 线路占用率 |
|----------|------|----------|-----------|
| **DMA 方式** | DMA 控制器自动从内存搬运数据到 UART TX FIFO | 极低（仅在包边界中断 CPU） | **≈ 100%** |
| **CPU-driven（非DMA）** | CPU 逐字节/逐包写入 UART TX 寄存器 | 高（每个字节都需 CPU 介入） | **取决于 CPU 调度** |

**DMA 方式**下，UART 硬件几乎不会空闲，有效吞吐量接近理论值。

**CPU-driven 方式**下，UART 发送端的数据供给完全依赖 CPU 的调度。如果 CPU 因为任务切换、中断嵌套、临界区等原因无法及时供数，UART 硬件就会出现空闲间隙，导致**有效吞吐量等比例下降**。

### 3.2 "50% CPU 时间"的量化影响

假设系统采用 CPU-driven 传输，且由于任务调度、中断处理等因素，UART 发送任务实际只能获得 **50% 的 CPU 时间窗口**：

```
有效吞吐量 = 理论吞吐量 × 软件有效传输占空比
           = 80～85 KB/s × 50%
           ≈ 40～42.5 KB/s
```

在此条件下，各编码格式的表现：

| 编码格式 | 实际需求 | 有效带宽（50% 占空比） | 结果 |
|----------|----------|------------------------|------|
| SBC（328kbps） | ~45 KB/s | 40～42.5 KB/s | ❌ 非常极限，容易卡顿 |
| AAC（256kbps） | ~35 KB/s | 40～42.5 KB/s | ⚠️ 勉强可跑 |
| aptX（352kbps） | ~48 KB/s | 40～42.5 KB/s | ❌ 基本不够 |
| aptX HD / LDAC | >72 KB/s | 40～42.5 KB/s | ❌ 完全不行 |

> **核心观点**：波特率数字只是理论上限，**真正可用带宽必须乘以"软件有效传输占空比"**。

### 3.3 工程实践中的优化方向

如果必须使用 CPU-driven 传输，应确保以下措施至少落实一项：

1. **使用 DMA**：彻底绕过 CPU 占空比问题，推荐首选方案。
2. **提升 UART TX 中断/任务优先级**：确保 HCI 数据发送的 ISR 或 Task 优先级足够高，且不会被长时间关中断。
3. **增大 HCI ACL 包长**：每个 ACL 包携带接近 1021 字节的 L2CAP 载荷，减少头部开销比例，提升批量传输效率。
4. **开启硬件流控（RTS/CTS）**：防止接收端缓冲区溢出导致的数据丢失。

如果以上措施均无法保证，即使跑 SBC，也建议将波特率提升至 **1.5 Mbps 甚至 3 Mbps**，为软件延迟留出缓冲。

## 四、常见认知误区澄清

### 误区一："CPU Usage 不是 100%，说明 CPU 不是瓶颈"

**错误。** CPU Usage 反映的是 CPU 执行指令的时间占比，**不能直接反映 UART 发送端的调度延迟**。

真正决定 UART 吞吐量的是 **"UART TX Task/ISR 能否获得足够长的连续时间窗口"**，而不是 CPU 是否空闲。

```
┌─────────────────────────────────────────────────────────────┐
│  反例：CPU Usage = 30%，但 UART 吞吐量很低                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [高优先级 Task]  ████░░░░░░████░░░░░░████░░░░░░           │
│  [UART TX Task]   ░░░░██░░░░░░██░░░░░░██░░░░░░             │
│  [CPU 空闲]       ░░░░░░██░░░░░░██░░░░░░██░░░░             │
│                                                             │
│  问题：UART Task 每次只运行很短时间就被切换                 │
│        虽然 CPU 有大量空闲，但 UART 线路被切成了碎片        │
│        有效吞吐量远低于理论值                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**正确的判断方法**：观察 UART TX 线路的实际占用率（用示波器或逻辑分析仪），或测量 UART TX Task 的调度延迟。

### 误区二："UART 是半双工，TX 和 RX 会互相抢占"

**错误。** HCI UART 通常是 **全双工（Full-Duplex）** 设计：

```
Host (MCU)          Controller (BT Chip)
    TX  ──────────────►  RX
    RX  ◄──────────────  TX
   RTS  ──────────────►  CTS
   CTS  ◄──────────────  RTS
```

TX 和 RX 使用独立的信号线，**可以同时收发，互不阻塞**。

A2DP 音频流主要是 **Host → Controller 的单向大数据流**，反向的 HCI Event/ACK 流量极小，对 TX 方向的影响可以忽略。因此，**不需要因为"半双工"而额外打折**。

### 误区三："同步传输"等同于"CPU 逐字节搬运"

**术语澄清。** 在 UART 语境下，"同步传输"（Synchronous Transmission）通常指带有时钟信号（CLK）的通信模式（如 SPI、I2S），与"异步传输"（Asynchronous，即标准 UART）相对。

本文讨论的"CPU 逐字节搬运"应准确表述为 **"CPU-driven（非DMA）传输"**，避免与 UART 的同步/异步模式混淆。

## 五、综合结论

### 5.1 波特率选择建议

| 使用场景 | 推荐波特率 | 说明 |
|----------|-----------|------|
| SBC / AAC / aptX | 921600 | 完全够用，余量充足 |
| aptX HD | 921600 或 1.5 Mbps | 921600 接近极限，建议留余量 |
| LDAC 660 kbps | **≥ 1.5 Mbps** | 921600 理论上限不足 |
| LDAC 990 kbps | **≥ 3 Mbps** 或 SDIO/USB | 必须升级接口 |

### 5.2 关键检查清单

在评估 UART 波特率是否足够时，建议按以下顺序排查：

- [ ] **确认编码格式及其实时码率** — 不同编码差异巨大
- [ ] **确认传输方式** — DMA 还是 CPU-driven？
- [ ] **确认 UART 线路实际占用率** — 用示波器/逻辑分析仪测量
- [ ] **确认 HCI TX Task/ISR 的优先级** — 是否会被频繁抢占？
- [ ] **确认是否开启硬件流控（RTS/CTS）** — 软件流控（XON/XOFF）在高速下不可靠
- [ ] **确认 HCI ACL 包长配置** — 尽量使用大包（接近 1021 字节）

## 六、多设备并发场景的波特率选择

实际产品中，蓝牙芯片通常不会只连接一个 A2DP 设备。以典型的 WiFi+BT Combo 芯片应用场景为例，系统可能同时维护以下连接：

| 设备类型 | 协议/Profile | 数据流向 | 典型流量特征 |
|----------|-------------|----------|-------------|
| 耳机/音箱 | A2DP + AVRCP | Host ↔ Controller | 连续大流量，实时性要求高 |
| 鼠标 | HID (BR/EDR 或 BLE) | Controller → Host | 间歇小流量，低延迟要求 |
| BLE 语音遥控器 | GATT/HID over LE | Controller → Host | 间歇小流量，语音激活时突发中等流量 |
| 键盘 | HID (BR/EDR 或 BLE) | Controller → Host | 间歇小流量，按键事件突发 |

**关键前提**：所有 Profile 的数据都复用同一条 HCI UART 物理链路，通过 HCI ACL 数据包的多路复用机制区分不同连接。

### 6.1 各设备流量估算

#### 常规 HID 设备

| 设备 | 典型场景 | 峰值数据率 | HCI 层开销后 |
|------|----------|-----------|-------------|
| HID Mouse | 高速移动 | ~1-2 KB/s | ~2-3 KB/s |
| HID Keyboard | 快速打字 | ~0.5-1 KB/s | ~1-2 KB/s |

#### BLE 语音遥控器

语音输入会显著增加 BLE 上行流量。假设语音编码后 64 kbps，通过 BLE Notify 传输：

| BLE MTU | 每包语音载荷 | 包率 | HCI 层总流量 |
|---------|-------------|------|-------------|
| 23 Bytes（默认） | 20 Bytes | ~400 包/秒 | ~12.4 KB/s |
| 185 Bytes（协商后） | 180 Bytes | ~45 包/秒 | ~8.6 KB/s |

**保守估算：BLE 语音遥控器在上行方向产生约 10～15 KB/s 的 HCI 流量。**

### 6.2 A2DP Source 场景（Host → Controller）

Source 场景下，A2DP 音频数据从 Host 流向 Controller，同时 Controller 向 Host 上报 HID/BLE 事件。

**下行方向 (Host → Controller)：**

| 来源 | 流量 | 说明 |
|------|------|------|
| A2DP 音频 | 取决于编码格式 | 连续，见第二节表格 |
| HID 控制命令 | ~1 KB/s | 间歇 |
| HCI Command | ~2 KB/s | 间歇 |

**上行方向 (Controller → Host)：**

| 来源 | 流量 | 说明 |
|------|------|------|
| HID Mouse 报告 | ~2 KB/s | 间歇 |
| HID Keyboard 报告 | ~1 KB/s | 间歇 |
| BLE 语音数据 | ~12 KB/s | 突发（语音激活时） |
| HCI Event | ~2 KB/s | 间歇 |
| **上行总计** | **~17 KB/s** | 语音激活时 |

**Source 场景波特率选择矩阵：**

| 使用场景 | 下行流量 | 上行流量 | 最低波特率 | 推荐波特率 |
|----------|----------|----------|-----------|-----------|
| SBC + HID + BLE 语音 | ~53 KB/s | ~17 KB/s | 921600 | **921600** |
| aptX + HID + BLE 语音 | ~56 KB/s | ~17 KB/s | 921600 | **1.5 Mbps** |
| aptX HD + HID + BLE 语音 | ~88 KB/s | ~17 KB/s | 1.5 Mbps | **3 Mbps** |
| LDAC 660k + 任何设备 | ~95 KB/s | ~17 KB/s | 1.5 Mbps | **3 Mbps** |
| LDAC 990k | ~140 KB/s | ~17 KB/s | 3 Mbps | **SDIO/USB** |

> **分析**：多设备并发时，瓶颈仍然在下行方向的 A2DP 音频流量。上行方向的 HID/BLE 流量总和约 17 KB/s，远低于 921600 的理论上限。但需注意：如果 Controller 侧 UART TX 也是 CPU-driven，Controller 内部任务繁忙时可能导致 HID 事件延迟。

### 6.3 A2DP Sink 场景（Controller → Host）

Sink 场景下，Controller 从蓝牙空口接收音频数据，再通过 UART 上行传输给 Host 解码输出。

**关键差异**：

| 特性 | A2DP Source | A2DP Sink |
|------|------------|-----------|
| 音频数据方向 | Host → Controller | Controller → Host |
| CPU-driven 侧 | Host 侧 TX | **Controller 侧 TX** |
| 瓶颈位置 | Host 侧 CPU 供数 | **Controller 侧 CPU 供数** |

**Sink 场景的风险更高：**

1. **Controller 侧 CPU 更繁忙** — 需要同时处理蓝牙基带、WiFi、多设备连接维护
2. **Controller 侧 CPU 性能通常较弱** — 相比 Host（手机/平板应用处理器）
3. **CPU-driven 传输的占空比损耗可能更大** — Controller 内部任务抢占更激烈

**Sink 场景流量估算：**

| 来源 | 流量 | 说明 |
|------|------|------|
| A2DP Sink 音频 | 取决于编码格式 | 与 Source 对称，方向相反 |
| HID Mouse 报告 | ~2 KB/s | 间歇 |
| HID Keyboard 报告 | ~2 KB/s | 间歇 |
| BLE 语音数据 | ~12 KB/s | 突发 |
| HCI Event | ~2 KB/s | 间歇 |
| **上行总计** | **~61 KB/s (SBC)** | 语音激活时 |

**Sink 场景波特率选择矩阵：**

| 使用场景 | 关键流量 | 最低波特率 | 推荐波特率 | 风险等级 |
|----------|----------|-----------|-----------|----------|
| SBC + HID + BLE 语音 | ~61 KB/s 上行 | 921600 | **1.5 Mbps** | 中 |
| aptX + HID + BLE 语音 | ~64 KB/s 上行 | 1.5 Mbps | **1.5 Mbps** | 中 |
| aptX HD + HID + BLE 语音 | ~94 KB/s 上行 | 1.5 Mbps | **3 Mbps** | 高 |
| LDAC 660k + 任何设备 | ~102 KB/s 上行 | 3 Mbps | **3 Mbps** | 高 |
| LDAC 990k | ~147 KB/s 上行 | 3 Mbps | **SDIO/USB** | 极高 |

> **为什么 Sink 推荐波特率更高？** Controller 侧作为数据发送方，其 CPU 需要同时处理蓝牙基带、WiFi 协议栈、多连接维护等任务。CPU-driven 传输下，Controller 内部的任务竞争更激烈，有效吞吐量折扣可能从 80-90% 降至 60-80%。因此建议在 Source 推荐值基础上再留 20-30% 余量。

### 6.4 Sink 场景的特殊优化建议

1. **Controller 侧 UART TX ISR 优先级设为最高** — 确保音频数据不被内部任务延迟
2. **避免长时间关中断** — 任何临界区都会直接阻塞音频数据发送
3. **Controller 侧尽量使用 DMA** — 如果硬件支持，DMA 能显著缓解 Controller CPU 压力
4. **HCI ACL 包长协商最大化** — Sink 场景下 Controller 作为发送方，大包能减少包率开销

## 七、综合结论

### 7.1 波特率选择建议（完整版）

| 使用场景 | 主导方向 | 关键流量 | 最低波特率 | 推荐波特率 |
|----------|----------|----------|-----------|-----------|
| **Source: SBC/AAC/aptX** | Host → Controller | ~50 KB/s | 921600 | **921600** |
| **Source: aptX HD** | Host → Controller | ~78 KB/s | 921600 | **1.5 Mbps** |
| **Source: LDAC 660k** | Host → Controller | ~90 KB/s | 1.5 Mbps | **3 Mbps** |
| **Source: LDAC 990k** | Host → Controller | ~135 KB/s | 3 Mbps | **SDIO/USB** |
| **Sink: SBC + 多 HID** | Controller → Host | ~61 KB/s | 921600 | **1.5 Mbps** |
| **Sink: aptX + 多 HID** | Controller → Host | ~64 KB/s | 1.5 Mbps | **1.5 Mbps** |
| **Sink: aptX HD + 多 HID** | Controller → Host | ~94 KB/s | 1.5 Mbps | **3 Mbps** |
| **Sink: LDAC 660k + 多 HID** | Controller → Host | ~102 KB/s | 3 Mbps | **3 Mbps** |
| **Sink: LDAC 990k** | Controller → Host | ~147 KB/s | 3 Mbps | **SDIO/USB** |

### 7.2 关键检查清单

在评估 UART 波特率是否足够时，建议按以下顺序排查：

- [ ] **确认 A2DP 角色** — Source 还是 Sink？Sink 场景风险更高
- [ ] **确认编码格式及其实时码率** — 不同编码差异巨大
- [ ] **确认并发设备清单** — 鼠标、键盘、BLE 语音遥控器等都会占用带宽
- [ ] **确认传输方式** — DMA 还是 CPU-driven？Sink 场景下 Controller 侧尤为关键
- [ ] **确认 UART 线路实际占用率** — 用示波器/逻辑分析仪测量
- [ ] **确认 HCI TX Task/ISR 的优先级** — 是否会被频繁抢占？
- [ ] **确认是否开启硬件流控（RTS/CTS）** — 软件流控（XON/XOFF）在高速下不可靠
- [ ] **确认 HCI ACL 包长配置** — 尽量使用大包（接近 1021 字节）

### 7.3 一句话总结

> **921600 波特率跑普通 A2DP Source（SBC/AAC/aptX）完全够用；跑 aptX HD 接近极限；跑 LDAC 必须升级波特率。A2DP Sink 场景因 Controller 侧 CPU 压力更大，建议在 Source 推荐值基础上再留 20-30% 余量。无论哪种场景，DMA 传输都是确保稳定性的最佳实践。**
