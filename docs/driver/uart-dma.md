# UART DMA 传输机制详解

在嵌入式系统中，UART 是最常用的串行通信接口之一。当数据量较大或实时性要求较高时，**CPU 逐字节搬运数据**的方式会成为系统瓶颈。DMA（Direct Memory Access，直接存储器访问）技术通过让硬件自动完成数据传输，显著降低 CPU 负载，提升系统效率。

本文从 DMA 基础概念出发，深入分析 UART TX/RX 方向的 DMA 工作机制，并针对蓝牙 HCI 等变长包场景，给出最佳实践建议。

---

## 一、DMA 基础概念

### 1.1 什么是 DMA

DMA 是一种**无需 CPU 介入**的数据传输机制。DMA 控制器作为独立的硬件模块，可以在内存与外设之间、或内存与内存之间自动搬运数据。

```
┌─────────────────────────────────────────────────────────────┐
│                  数据传输方式对比                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  CPU-driven 方式：                                           │
│  ┌─────┐    中断/轮询    ┌─────────┐    逐字节读写    ┌─────┐│
│  │ CPU │◄───────────────►│ UART    │◄───────────────►│ FIFO││
│  └─────┘                 │ 寄存器  │                  └─────┘│
│       ↑                                                  │
│       └──────────────────────────────────────────────────┘
│       每个字节都需要 CPU 介入，占用大量 CPU 时间            │
│                                                             │
│  DMA 方式：                                                  │
│  ┌─────┐   配置一次        ┌─────────┐   硬件自动    ┌─────┐ │
│  │ CPU │──────────────────►│  DMA    │─────────────►│ FIFO│ │
│  └─────┘   后续无需介入     │ 控制器  │   搬运数据    └─────┘ │
│       ↑                                                  │
│       └──── 仅在传输完成/异常时中断 CPU ──────────────────┘
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 为什么需要 DMA

| 场景 | CPU-driven 的问题 | DMA 的优势 |
|------|------------------|-----------|
| 高速 UART（921600+） | CPU 频繁中断，吞吐量受限 | 硬件自动搬运，吞吐量接近理论值 |
| 大数据块传输 | CPU 长时间被占用，无法执行其他任务 | CPU 仅在边界介入，并行处理其他任务 |
| 实时性要求高的系统 | 中断延迟导致数据丢失 | 减少中断频率，降低延迟抖动 |
| 低功耗系统 | CPU 频繁唤醒，功耗高 | CPU 可进入休眠，DMA 独立完成传输 |

### 1.3 DMA 的核心参数

DMA 控制器工作时，需要以下关键参数（由 CPU 在初始化时配置）：

| 参数 | 含义 | 谁来设置 |
|------|------|----------|
| **源地址（Source Address）** | 数据来源的内存地址或外设寄存器地址 | CPU 初始化 |
| **目的地址（Destination Address）** | 数据去向的内存地址或外设寄存器地址 | CPU 初始化 |
| **传输长度（Transfer Length）** | 需要传输的总字节数 | CPU 初始化 |
| **传输方向** | 内存→外设 / 外设→内存 / 内存→内存 | CPU 初始化 |
| **触发源** | 什么条件触发 DMA 开始搬运 | CPU 初始化 |

> **关键认知**：DMA 控制器是硬件逻辑，一旦配置完成，就按照这些参数自动工作，不需要 CPU 再逐字节干预。

## 二、UART TX + DMA（发送方向）

### 2.1 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                 UART TX + DMA 工作流程                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  初始化阶段（CPU 执行一次）：                                 │
│  ├── 配置 DMA 源地址: 发送缓冲区（内存）                     │
│  ├── 配置 DMA 目的地址: UART TX FIFO 寄存器                  │
│  ├── 配置传输长度: 待发送数据长度                            │
│  └── 配置触发源: UART TX FIFO 空/可写                       │
│                                                             │
│  运行阶段（DMA 自动工作）：                                   │
│                                                             │
│  内存 TX Buffer ──► DMA 控制器 ──► UART TX FIFO ──► 线路   │
│       ↑                                              │      │
│       │         硬件自动搬运，无需 CPU                  │      │
│       │                                              │      │
│       └────────────── 传输完成中断 ────────────────────┘      │
│                        （CPU 只需处理边界）                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键特性

| 特性 | 说明 |
|------|------|
| **内存地址递增** | DMA 从 TX Buffer 逐字节读取，源地址自动递增 |
| **外设地址固定** | UART TX FIFO 寄存器地址不变，目的地址固定 |
| **触发条件** | UART TX FIFO 有空位时，DMA 自动搬运下一字节 |
| **中断时机** | 传输完成时中断 CPU，通知可以准备下一包数据 |

### 2.3 与 CPU-driven TX 的对比

| 维度 | CPU-driven | DMA |
|------|-----------|-----|
| CPU 占用 | 高（每字节进一次 ISR） | 极低（仅在包边界中断） |
| 吞吐量 | 受 CPU 调度影响 | 接近理论值 |
| 线路占用率 | 取决于 CPU 供数能力 | ≈ 100% |
| 适用场景 | 低速、简单系统 | 高速、实时系统 |

## 三、UART RX + DMA（接收方向）

### 3.1 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                 UART RX + DMA 工作流程                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  初始化阶段（CPU 执行一次）：                                 │
│  ├── 配置 DMA 源地址: UART RX FIFO 寄存器                    │
│  ├── 配置 DMA 目的地址: 内存 RX Buffer                       │
│  ├── 配置传输长度: 缓冲区大小（如 1024）                     │
│  ├── 配置触发源: UART RX FIFO 非空                          │
│  └── 配置模式: 循环模式（Circular Mode）                     │
│                                                             │
│  运行阶段（DMA 自动工作）：                                   │
│                                                             │
│  线路 ──► UART RX FIFO ──► DMA 控制器 ──► 内存 RX Buffer   │
│                              │                      │       │
│                              │    硬件自动搬运       │       │
│                              │                      │       │
│                              └────── 中断触发 ───────┘       │
│                                     （CPU 处理边界）         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 循环模式（Circular Mode）

RX 方向通常使用**循环模式**，因为数据是持续流入的，没有明确的"结束"。

```
┌─────────────────────────────────────────────────────────────┐
│              DMA 循环模式（环形缓冲区）                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  内存 RX Buffer（1024 字节）：                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  [0]  [1]  [2]  ...  [510]  [511]  [512]  ... [1023] │   │
│  │   ▲                                          ▲        │   │
│  │   │                                          │        │   │
│  │   └────────── 已用数据（软件可读）───────────┘        │   │
│  │                                                             │
│  │   ▲                                          ▲        │   │
│  │   │                                          │        │   │
│  │   └────────── 空闲空间（DMA 可写）───────────┘        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  DMA 写指针：硬件自动维护，循环递增                          │
│  CPU 读指针：软件维护，循环递增                              │
│                                                             │
│  当写指针到达 [1023] 后，自动回到 [0] 继续写入               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 双指针机制

RX 方向 DMA 涉及两个独立的指针：

| 指针 | 谁维护 | 谁读取 | 谁写入 |
|------|--------|--------|--------|
| **写指针（Write Pointer）** | DMA 硬件 | CPU 软件（读 DMA 寄存器） | 只有 DMA |
| **读指针（Read Pointer）** | CPU 软件 | CPU 软件自己 | 只有 CPU |

```
┌─────────────────────────────────────────────────────────────┐
│                   双指针协作机制                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  写指针（DMA 维护）：                                        │
│  ├── DMA 每写入 1 字节，自动递增                             │
│  ├── CPU 可通过读取 DMA 当前地址寄存器获取                   │
│  └── DMA 不知道读指针的存在！                                │
│                                                             │
│  读指针（CPU 维护）：                                        │
│  ├── CPU 每读取 1 字节，软件递增                             │
│  ├── 只有 CPU 自己知道读指针位置                             │
│  └── CPU 不知道 DMA 何时会覆盖未读数据！                     │
│                                                             │
│  已用空间计算：                                              │
│  ├── 写指针 >= 读指针：已用 = 写 - 读                        │
│  └── 写指针 < 读指针：已用 = 总长 - 读 + 写（循环情况）      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

> **关键认知**：DMA 只管写，不管读；CPU 既要读，还要监控；两者没有直接的硬件同步机制，溢出风险需要软件管理。

## 四、固定长度 vs 变长包的处理

DMA 中断配置取决于数据包的特性：

### 4.1 固定长度数据（适合半传输中断）

```
应用场景：每次接收固定 1024 字节的数据块

DMA 配置：
├── 模式: 循环模式
├── 传输长度: 1024 字节
└── 中断: 半传输(512字节) + 传输完成(1024字节)

工作流程（双缓冲效果）：
┌─────────────────────────────────────────────────────────────┐
│  内存缓冲区:                                                 │
│  [0]...[511]│[512]...[1023]                                  │
│   前半区    │  后半区                                        │
│      ↑           ↑                                           │
│   半传输中断  传输完成中断                                    │
│      │           │                                           │
│      ▼           ▼                                           │
│  CPU 处理前半  CPU 处理后半                                   │
│  DMA 继续写后半 DMA 回到开头                                  │
│                                                             │
│  双缓冲效果：CPU 处理前半时，DMA 同时写入后半                 │
│            CPU 处理后半时，DMA 同时写入前半（循环）           │
└─────────────────────────────────────────────────────────────┘

半传输中断时 CPU 做什么？
└── 处理前半缓冲区数据（512字节），DMA 继续往后半区写

传输完成中断时 CPU 做什么？
└── 处理后半缓冲区数据（512字节），DMA 回到开头继续写
```

### 4.2 变长包数据（适合 IDLE 中断）

```
应用场景：HCI ACL 包，长度不固定

DMA 配置：
├── 模式: 循环模式
├── 传输长度: 1024（缓冲区大小，只是一个上限）
└── 中断: 关闭半传输/传输完成，只用 UART IDLE 中断

工作流程：
┌─────────────────────────────────────────────────────────────┐
│  UART 收到数据 ──► DMA 持续搬运到 Ring Buffer                │
│                                                             │
│  数据持续流入...                                             │
│                                                             │
│  UART 线路空闲（IDLE）──► IDLE 中断触发                      │
│       ↑                                                     │
│       │ 表示：一包数据接收完毕                               │
│       │                                                     │
│       ▼                                                     │
│  CPU 进入 ISR                                               │
│       │                                                     │
│       ▼                                                     │
│  读取 DMA 当前写入位置（如 750）                             │
│       │                                                     │
│       ▼                                                     │
│  通知上层：新数据从 [上次位置] 到 [750]                      │
│                                                             │
│  没有"半传输"概念，因为每包长度不固定                        │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 IDLE 中断的触发机制

```
IDLE 中断触发条件：
┌─────────────────────────────────────────────────────────────┐
│  1. UART 收到至少 1 字节数据（RX 非空）                      │
│  2. 随后线路空闲时间超过 1 帧时间（通常 1 位时间）           │
│                                                             │
│  两个条件必须同时满足，缺一不可！                            │
└─────────────────────────────────────────────────────────────┘

为什么不会持续产生？
┌─────────────────────────────────────────────────────────────┐
│  场景一：持续有数据流入                                      │
│  UART 状态: 一直在接收，RX 从未空闲                          │
│  IDLE 中断: ❌ 不触发（条件 2 不满足）                       │
│                                                             │
│  场景二：数据流结束，线路空闲                                │
│  UART 状态: 最后一字节接收完毕，空闲时间 > 1 帧时间          │
│  IDLE 中断: ✅ 触发一次（条件 1 和 2 都满足）                │
│                                                             │
│  场景三：一直没有任何数据                                    │
│  UART 状态: RX 为空，从未收到过数据                          │
│  IDLE 中断: ❌ 不触发（条件 1 不满足）                       │
└─────────────────────────────────────────────────────────────┘

中断清除：
├── 触发后，硬件自动置位 IDLE 标志
├── 软件需要读取 UART DR（数据寄存器）清除标志
└── 清除后，等待下一次"收到数据后空闲"才会再次触发
```

> **关键认知**：IDLE 中断检测的是"一包数据结束"的事件，而不是"线路空闲"的状态。一直无数据不会触发，数据持续流入也不会触发。

### 4.3 两种模式的对比

| 特性 | 固定长度（半传输中断） | 变长包（IDLE 中断） |
|------|----------------------|-------------------|
| 适用数据 | 固定块大小 | 协议包（如 HCI） |
| 中断时机 | 到达预设长度的一半/全部 | 线路空闲超时 |
| CPU 处理粒度 | 按固定块处理 | 按实际包长度处理 |
| 双缓冲效果 | 天然支持 | 需要软件实现 |
| 典型应用 | 音频采样数据、传感器数据 | 蓝牙 HCI、AT 指令 |

## 五、Ring Buffer 与溢出防护

### 5.1 溢出的可能性

由于 DMA 不知道读指针的存在，如果 CPU 处理太慢，DMA 会覆盖未读数据：

```
┌─────────────────────────────────────────────────────────────┐
│                    溢出场景分析                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  场景：CPU 处理太慢，DMA 写满了整个缓冲区                     │
│                                                             │
│  时间线:                                                     │
│                                                             │
│  T0:  写指针=0, 读指针=0  （空）                            │
│       │                                                     │
│  T1:  DMA 写入 100 字节                                     │
│       写指针=100, 读指针=0                                  │
│       │                                                     │
│  T2:  DMA 写入 500 字节                                     │
│       写指针=600, 读指针=0                                  │
│       │                                                     │
│  T3:  CPU 处理了 200 字节                                   │
│       写指针=600, 读指针=200                                │
│       │                                                     │
│  T4:  DMA 持续写入，CPU 没处理                              │
│       写指针=1023, 读指针=200                               │
│       │                                                     │
│  T5:  DMA 继续写，回到开头（循环模式）                      │
│       写指针=199, 读指针=200                                │
│       ⚠️ 写指针即将追上读指针！                              │
│       │                                                     │
│  T6:  DMA 再写 1 字节                                       │
│       写指针=200, 读指针=200                                │
│       ❌ 溢出！新数据覆盖了未读数据！                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 溢出检测方法

```
// 软件定期检查已用空间
uint16_t write_pos = DMA_GetCurrentPosition();  // 读 DMA 寄存器
uint16_t read_pos = g_read_ptr;                  // 软件维护

uint16_t used_space;
if (write_pos >= read_pos) {
    used_space = write_pos - read_pos;
} else {
    used_space = BUFFER_SIZE - read_pos + write_pos;  // 循环情况
}

uint16_t free_space = BUFFER_SIZE - used_space;

if (free_space < THRESHOLD) {
    // 缓冲区即将满，告警或丢弃旧数据
    LOG_WARN("RX buffer nearly full! Used: %d, Free: %d", used_space, free_space);
}
```

### 5.3 防溢出策略

| 策略 | 机制 | 效果 |
|------|------|------|
| **大缓冲区** | 给 CPU 足够的处理时间窗口 | 降低溢出概率 |
| **硬件流控（RTS/CTS）** | 接收方拉低 RTS 通知发送方暂停 | 最后一道防线，防止数据丢失 |
| **软件监控** | 定期检查 used_space/free_space | 及时发现并告警 |
| **优先级优化** | 确保 CPU 处理 Task/ISR 优先级足够高 | 减少处理延迟 |

```
RTS/CTS 流控机制：
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Host RTS ──────► Controller CTS                            │
│                                                             │
│  当 Host RX Buffer 即将满时:                                │
│  ├── Host 拉低 RTS（通知 Controller 停止发送）               │
│  ├── Controller 检测到 CTS 低 → 暂停 UART TX                 │
│  └── 给 Host CPU 时间处理积压数据                            │
│                                                             │
│  当 Host 处理完数据后:                                      │
│  ├── Host 拉高 RTS（通知 Controller 恢复发送）               │
│  └── Controller 继续发送                                     │
│                                                             │
│  这是硬件自动的，不需要软件逐字节控制！                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 六、HCI 数据的最佳实践

### 6.1 为什么 HCI 需要特殊处理

蓝牙 HCI 数据具有以下特性：

| 特性 | 影响 |
|------|------|
| **变长包** | ACL 包长度不固定，无法使用半传输中断 |
| **持续流** | 数据持续流入，没有明确的"结束" |
| **实时性** | 音频数据对延迟敏感 |
| **多路复用** | 多个连接的数据复用同一条 UART |

### 6.2 推荐配置

```
UART RX + DMA 推荐配置（HCI 场景）：

DMA 配置:
├── 模式: 循环模式（Circular）
├── 触发: UART RX FIFO 非空即搬运
├── 传输长度: 2048 或更大（缓冲区大小）
└── 中断: 关闭半传输/传输完成

UART 配置:
├── 波特率: 根据实际带宽需求选择（参考 A2DP 波特率指南）
├── 数据位: 8 位
├── 停止位: 1 位
├── 校验: 无
├── 流控: RTS/CTS 硬件流控（必须开启）
├── 使能 RX DMA
└── 使能 IDLE 中断（线路空闲检测）

软件架构:
├── IDLE ISR: 计算新数据范围，通知解析任务
├── HCI 解析任务: 从 Ring Buffer 提取完整 HCI 包
└── 缓冲区监控: 定期检查 used_space，告警溢出风险
```

### 6.3 伪代码示例

// ========== 初始化 ==========
void uart_rx_dma_init(void) {
    // 1. 配置 DMA
    dma_config_t dma_cfg = {
        .src_addr = UART_DR_ADDRESS,      // UART 数据寄存器
        .dst_addr = (uint32_t)rx_buffer,  // 内存 RX Buffer
        .length = RX_BUFFER_SIZE,         // 2048
        .direction = DMA_PERIPH_TO_MEM,   // 外设到内存
        .mode = DMA_MODE_CIRCULAR,        // 循环模式
        .src_inc = DMA_INC_DISABLE,       // 外设地址不递增
        .dst_inc = DMA_INC_ENABLE,        // 内存地址递增
    };
    dma_init(DMA_CHANNEL_RX, &dma_cfg);
    
    // 2. 配置 UART
    uart_config_t uart_cfg = {
        .baudrate = 921600,
        .data_bits = UART_DATA_8BITS,
        .stop_bits = UART_STOP_1BIT,
        .parity = UART_PARITY_NONE,
        .flow_ctrl = UART_HW_FLOWCTRL_RTS_CTS,  // 硬件流控
        .use_rx_dma = true,
    };
    uart_init(UART_INSTANCE, &uart_cfg);
    
    // 3. 使能 UART IDLE 中断
    uart_enable_idle_interrupt(UART_INSTANCE);
    
    // 4. 启动 DMA
    dma_start(DMA_CHANNEL_RX);
}

// ========== IDLE 中断处理 ==========
void uart_idle_isr(void) {
    // 清除 IDLE 标志（必须先读 SR 再读 DR）
    uint32_t sr = uart_read_status(UART_INSTANCE);
    uint32_t dr = uart_read_data(UART_INSTANCE);
    (void)dr;  // 清除标志
    
    // 获取 DMA 当前写入位置
    uint16_t current_write_pos = dma_get_current_position(DMA_CHANNEL_RX);
    
    // 计算本次接收的数据范围
    if (current_write_pos != last_read_pos) {
        // 通知 HCI 解析任务处理新数据
        hci_rx_notify(last_read_pos, current_write_pos);
        
        // 更新读指针（注意：实际读取由解析任务完成，这里只是记录）
        // 或者使用消息队列通知解析任务
    }
}

// ========== HCI 解析任务 ==========
void hci_rx_task(void) {
    while (1) {
        // 等待 IDLE 中断通知
        rx_event_t evt;
        queue_receive(rx_queue, &evt, portMAX_DELAY);
        
        // 从 Ring Buffer 读取数据
        uint16_t read_pos = evt.start_pos;
        uint16_t write_pos = evt.end_pos;
        
        while (read_pos != write_pos) {
            // 读取 HCI 包头（4 字节）
            hci_acl_header_t header;
            ring_buffer_read(&header, sizeof(header), &read_pos);
            
            // 根据长度读取完整包
            uint16_t pkt_len = header.length;
            uint8_t *pkt = malloc(pkt_len);
            ring_buffer_read(pkt, pkt_len, &read_pos);
            
            // 分发到上层处理
            hci_dispatch_packet(pkt, pkt_len);
            free(pkt);
        }
        
        // 更新全局读指针
        last_read_pos = write_pos;
    }
}

// ========== 溢出监控（可选） ==========
void rx_buffer_monitor(void) {
    uint16_t write_pos = dma_get_current_position(DMA_CHANNEL_RX);
    uint16_t read_pos = last_read_pos;
    
    uint16_t used_space = (write_pos >= read_pos) 
        ? (write_pos - read_pos) 
        : (RX_BUFFER_SIZE - read_pos + write_pos);
    
    uint16_t usage_percent = (used_space * 100) / RX_BUFFER_SIZE;
    
    if (usage_percent > 80) {
        LOG_WARN("RX buffer usage: %d%%, risk of overflow!", usage_percent);
    }
}

## 七、常见问题与误区

### Q1：DMA 配置后还需要 CPU 参与数据传输吗？

**不需要逐字节参与，但需要在边界介入。**

- 数据传输过程：DMA 自动完成，CPU 不参与
- 传输完成/半传输：DMA 中断 CPU，CPU 处理边界逻辑
- 异常处理：CPU 处理 DMA 错误中断

### Q2：RX 方向用 DMA 后，CPU 还需要处理中断吗？

**需要，但中断频率大大降低。**

| 方式 | 中断频率 | CPU 工作量 |
|------|----------|-----------|
| 逐字节中断 | 每字节 1 次 | 极高 |
| FIFO 阈值中断 | 每 16 字节 1 次 | 中等 |
| DMA + IDLE 中断 | 每包 1 次 | 极低 |

### Q3：DMA 的写指针和 CPU 的读指针如何同步？

**没有硬件同步，需要软件管理。**

- CPU 通过读取 DMA 寄存器获取写指针
- CPU 自己维护读指针
- 两者没有直接的硬件握手
- 溢出风险需要软件监控和流控来防范

### Q4：IDLE 中断会持续触发吗？

**不会。**

- 触发条件：先收到数据，再空闲超时
- 一直无数据：不触发（未满足"先收到数据"）
- 数据持续流入：不触发（未满足"空闲超时"）
- 触发后：需要软件清除标志，等待下一次"收到数据后空闲"

### Q5：半传输中断和 IDLE 中断可以同时用吗？

**不建议。**

- 半传输中断适用于固定长度数据
- IDLE 中断适用于变长包
- 两者逻辑不同，混用会导致混乱
- HCI 变长包场景只用 IDLE 中断即可

## 八、总结

| 要点 | 结论 |
|------|------|
| DMA 的本质 | 硬件自动搬运数据，CPU 只在边界介入 |
| TX 方向 | DMA 从内存搬运到 UART FIFO，传输完成中断 |
| RX 方向 | DMA 从 UART FIFO 搬运到内存，推荐循环模式 |
| 固定长度数据 | 用半传输中断，双缓冲并行处理 |
| 变长包数据 | 用 IDLE 中断，按包边界处理 |
| 溢出防护 | 大缓冲区 + 硬件流控 + 软件监控 |
| HCI 最佳实践 | 循环模式 + IDLE 中断 + RTS/CTS 流控 |

> **核心原则：DMA 只管搬运，不管协议；CPU 只管边界，不管字节；流控是最后一道防线。**
